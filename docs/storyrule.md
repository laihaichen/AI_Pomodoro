# 模拟人生游戏规则

## 1. 核心机制

⚠️ **概念区分**：
- **原始随机数**：后台自动生成的1-100整数，随每次推进番茄钟刷新，是命运值公式的输入
- **健康度**：满分10分，游戏开始时为9分。健康度由 **陪审团系统** 独立管控（见下文），AI 不参与健康度判定。健康度只减不增
- **吉凶值**：后台根据健康度概率函数自动计算，+1（吉）或-1（凶），与原始随机数完全独立
- **命运值**：经过完整计算后的最终结果，用于触发事件
- **超时惩罚随机数R**：从面板「超时惩罚随机数」字段直接读取，后端已自动生成

**核心流程**：
- **原始随机数**：后台 `random.randint(1, 100)` 自动生成，推进番茄钟时刷新
- **吉凶值计算**（后台自动，与原始随机数完全独立）：
  - 若时间间隔 ≥ 15分钟 → 强制判**凶（-1）**
  - 若时间间隔 < 15分钟 → 以健康度为参数独立概率抽取：`random.random() < health/10` → True 则**吉（+1）**，False 则**凶（-1）**
- **命运值计算**：`命运值 = 原始随机数 × 吉凶值 - 超时惩罚随机数R`（后台自动完成，写入面板）
- **年龄系统**：记录条数"X"对应角色年龄
  - 明确约束：第1条记录即对应1岁，角色年龄必须从1岁开始按记录数逐年增长，不允许从0岁或其他年龄起步
- **输入格式**：AI从游戏状态面板直接读取「最终命运值」「吉凶结果」「健康度」，无需用户手工填写
- **学习助手系统**：玩家可在游戏开始前选择携带技能的学习助手。助手的技能可能在后台自动修改游戏状态。AI 无需了解具体助手和技能细节，只需照常从面板读取最终值即可。当前助手可能修改的状态包括：
  - 最终命运值（在后台计算完成后进一步叠加增减）
- **陪审团系统**：系统内置了一个独立的陪审团系统，由3名不在场的学习助手组成。详见下文「陪审团与进度指示器」节。

## 2. 健康度与吉凶判断规则

**健康度系统**：
- 游戏开始时健康度为 **9分**（满分10分，为未来增加机制保留空间）
- 健康度由 **陪审团系统** 独立管控，AI 不参与健康度判定
- 玩家自行决定何时将问题和回答递交给陪审团审议，陪审团投票结果决定健康度是否变化
- 健康度**只减不加**
- 健康度扣减由后台自动执行，无需玩家手动操作

**吉凶判断规则（后台自动计算，AI无需干预）**：
- 若时间间隔 ≥ 15分钟 → 强制判**凶（-1）**
- 若时间间隔 < 15分钟 → 以健康度为参数独立概率抽取：
  - `random.random() < health / 10` → True → **吉（+1）**
  - `random.random() < health / 10` → False → **凶（-1）**
  - 例：健康度=9 → 90%概率为吉；健康度=5 → 50%概率为吉
  - ⚠️ 此概率抽取与原始随机数**完全独立**，是两次不同的随机过程

## 3. 游戏流程

### 3.1 命运值处理
- 游戏状态面板已自动完成全部计算：原始随机数生成、吉凶值判定（概率独立）、超时惩罚、最终命运值
- AI从面板「最终命运值」字段直接读取命运值，**无需重新计算**
- AI从面板「吉凶结果」字段读取吉凶（吉/凶）
- AI从面板「健康度」字段读取当前健康度（由陪审团系统维护）
- 记录最终命运值

### 3.2 状态加载阶段（强制执行）

⚠️ **核心规则**：当角色年龄X > 1岁时，该角色的故事必须来自于角色X-1岁时JSON预设事件中「应该加载的预设事件，上一轮的」字段所指定区间对应的事件文本。

**「应该加载的预设事件，上一轮的」字段说明**：
- 这是游戏状态面板中的一个字段，由后台根据本轮最终命运值自动计算并写入
- 格式例如：`POS_MID(50~84) 中等正面事件` / `NEG_HIGH(-89~-60) 严重负面事件`
- 含义：**本轮命运值落在该区间，因此应从上一轮 JSON 中加载该区间对应的预设事件**
- AI **无需自己计算**命运值区间，直接读取该字段即可

在处理本轮模拟人生游戏前，必须执行 **[状态加载]** 动作：

1. **读取面板**：从「应该加载的预设事件，上一轮的」字段读取当前区间标签（如 `POS_MID`）
2. **回溯扫描**：读取上一轮回复末尾的 `JSON` 代码块
3. **强制提取**：从 JSON 的 `event_registry` 中提取面板标签对应区间的事件文本
4. **执行校验**：在输出"本轮命运值"后，必须显式输出一行：`📌 系统日志：已加载上一轮 JSON 预设事件 —— [区间键名，如 POS_HIGH]：[提取到的文本]`
5. **撰写故事**：基于该提取文本撰写故事，**严禁**更改或重新创作

⚠️ **特殊情况处理**：
- 角色1岁时：面板显示「当前为第一条记录，没有预设事件」→ 无需加载，直接生成与命运值相符的初始事件
- 新角色接管时：新角色第一条记录无需加载（前角色已死亡，JSON预判失效），直接生成与命运值相符的新角色初始事件
- 使用宿命卡时：跳过JSON加载，直接使用玩家指定的宿命事件

### 3.3 故事叙述
- 根据加载的事件文本（或角色1岁的初始事件、或宿命卡指定事件）讲述故事
- 故事应反映角色的成长轨迹和当前境遇
- **严禁偏离加载的事件文本**，必须忠实呈现预设内容

### 3.4 事件预判系统（JSON 状态锚点法）

**命运值范围与事件等级映射**：
- (-100 ~ -90)：FAIL(-100 ~ -90) - 失败事件（AI生成，游戏失败）
- (-89 ～ -60)：NEG_HIGH(-89 ~ -60) - 严重负面事件
- (-60 ~ -30)：NEG_MID(-60 ~ -30) - 中等负面事件
- (-30 ~ -1)：NEG_LOW(-30 ~ -1) - 轻度负面事件
- (1 ~ 49)：POS_LOW(1 ~ 49) - 轻度正面事件
- (50 ~ 84)：POS_MID(50 ~ 84) - 中等正面事件
- (85 ~ 100)：POS_HIGH(85 ~ 100) - 高度正面事件

**重要说明**：所有事件内容必须符合玩家在游戏开始时设定的故事类型

**输出规则**：

在每一轮回复的**最末尾**，AI必须输出一个 JSON 代码块，用于锁定下一年龄的所有预判事件。

**JSON 格式规范**：
```json
{
  "next_age": [下一年龄数字],
  "event_registry": {
    "FAIL(-100 ~ -90)": "[失败事件文本]",        // 命运值 -100 ~ -90
    "NEG_HIGH(-89 ~ -60)": "[严重负面事件文本]",  // 命运值 -89 ~ -60
    "NEG_MID(-60 ~ -30)": "[中等负面事件文本]",   // 命运值 -60 ~ -30
    "NEG_LOW(-30 ~ -1)": "[轻度负面事件文本]",   // 命运值 -30 ~ -1
    "POS_LOW(1 ~ 49)": "[轻度正面事件文本]",   // 命运值 1 ~ 49
    "POS_MID(50 ~ 84)": "[中等正面事件文本]",   // 命运值 50 ~ 84
    "POS_HIGH(85 ~ 100)": "[高度正面事件文本]"   // 命运值 85 ~ 100
  }
}
```

**关键要求**：
- JSON 代码块必须在每轮输出的**最后**
- 所有 7 个事件槽位必须填写完整的事件描述，**禁止留空或使用占位符**
- 事件文本应简洁明确（30-50字），便于后续加载和执行
- 如玩家使用了干预卡，需将玩家自定义的事件文本写入对应槽位

**触发逻辑**：

- **角色1岁时（首轮例外）**：
  - 用户提供原始随机数，AI计算命运值
  - 直接生成与该命运值等级相符的初始事件
  - 在输出末尾附上首个 JSON 代码块

- **角色年龄 ≥ 2岁时**：
  1. 执行[状态加载]阶段（见上文）
  2. 从上一轮JSON提取对应事件
  3. 输出系统日志确认加载
  4. 基于提取的事件文本撰写故事
  5. 在输出末尾附上新的 JSON 代码块

**触发示例**：

角色14岁输出的 JSON：
```json
{
  "next_age": 15,
  "event_registry": {
    "FAIL(-100 ~ -90)": "实验爆炸，身受重伤而亡",              // 命运值 -100 ~ -90
    "NEG_HIGH(-89 ~ -60)": "法术失控，烧毁了导师的实验室",    // 命运值 -89 ~ -60
    "NEG_MID(-60 ~ -30)": "材料调配失误，浪费大量珍贵药材",   // 命运值 -60 ~ -30
    "NEG_LOW(-30 ~ -1)": "魔力枯竭，需要休养三日",          // 命运值 -30 ~ -1
    "POS_LOW(1 ~ 49)": "成功绘制基础法阵",               // 命运值 1 ~ 49
    "POS_MID(50 ~ 84)": "调配出稳定的魔力药剂",            // 命运值 50 ~ 84
    "POS_HIGH(85 ~ 100)": "研究突破，成功制作二环法术'隐形术'卷轴"  // 命运值 85 ~ 100
  }
}
```

角色15岁时：
- 用户提交原始随机数，AI计算得命运值98
- AI回溯读取上述JSON，锁定 POS_HIGH(85 ~ 100) 区间
- 输出：`📌 系统日志：已加载上一轮 JSON 预设事件 —— POS_HIGH(85 ~ 100)：研究突破，成功制作二环法术'隐形术'卷轴`
- 基于该文本撰写故事
- 输出新的 JSON 代码块

## 4. 故事设定与宿命事件系统

### 4.1 故事类型设定（永久不变）
- 玩家在游戏开始时必须设定故事类型，简要描述故事的主题和风格
- 故事类型示例：
  - "这是一个关于核科学家的传记类故事"
  - "这是一个关于一个画家遭遇奇遇，长期修练和获得超凡力量的奇幻故事"
  - "这是一个关于创业者从零开始建立商业帝国的励志故事"
- **重要规则**：故事类型一旦设定，永远不会发生变化，所有事件都必须符合故事类型

### 4.2 失败事件规则（AI接管）
- 当命运值落入 FAIL(-100 ~ -90) 区间时，由 AI 根据当前故事发展自动生成失败事件
- **硬性规定**：失败事件必须与设定的故事类型相符合，且必须以角色死亡为结局（死亡是唯一允许的失败形式）
- 当失败事件发生后，游戏立即结束，代表游戏失败
- AI 生成的失败事件应与角色当前境遇、故事脉络保持连贯

### 4.3 AI生成事件
- 对于非宿命事件范围的命运值，AI根据事件等级自由生成相应事件
- **所有AI生成的事件必须符合玩家设定的故事类型**
- 事件内容应与角色年龄、学习状态、故事类型和故事发展相关
- 保持故事的连贯性和逻辑性
- **故事世界合理性优先**：严禁以任何形式（包括直接引用、隐喻、调侃等）把"prompt正文"涉及的学习主题、术语或题目与故事内容关联；如出现即视为恶性叙述，必须改写为符合故事时代与世界观的元素（如三国故事中绝不可描述Rust Trait Bounds）。
- **正确案例**："曹操出征袁绍，你作为谋士鼓励曹操袭击乌巢"（与设定一致）
- **错误案例**："曹操使用Trait Bounds (T: Add + Display)战胜袁绍"（穿越设定、逻辑混乱，被视为无效描述）

## 5. 幸运系统

### 5.1 触发条件

幸运系统可通过以下两种方式触发：

**方式1：命运值触发**
- 当命运值达到90以上时，玩家可以从幸运操作列表中抽取一个幸运操作

**方式2：阶段性指标触发（仅限平衡难度及以上）**
- 当玩家选择【平衡难度】或【硬核难度】时，在每个18条记录倍数节点（18条、36条、54条...），AI宣布玩家达成阶段性最低任务指标后，立即触发幸运系统
- AI将要求玩家从幸运操作列表中选取一项作为奖励
- 这是对玩家成功完成阶段性目标的奖励机制

**通用规则**：
- **默认设定**：如果玩家不主动选择幸运操作，则视为本次自动跳过
- 幸运系统可在整个流程中多次触发，只要满足任一触发条件即可再次抽取新的幸运操作

### 5.2 幸运操作列表

- [干预卡count + 1：玩家获得1张干预卡，可在之后任意时机使用]
- [宿命卡count + 1：玩家获得1张宿命卡，可在之后任意年龄使用]
- [换取5积分（不拿卡）]：**仅当**玩家同时持有 ≥5 张宿命卡和 ≥2 张干预卡时可选。该操作由后台 Dashboard 按钮直接执行（+5 积分并消费本次幸运奖励），不经过 AI 对话流程。AI 感知到的效果等同于玩家跳过了本次幸运操作。

### 5.3 功能目的
- 允许用户在高命运值时介入剧情发展，选择最合适的幸运操作
- 通过干预卡或宿命卡，将玩家意图融入故事叙述
- 提升玩家对故事走向及角色成长的掌控感
- 当卡牌资源溢出时，换分选项避免了"食之无味弃之可惜"的体验问题

### 5.4 干预卡机制

**基本规则**：
- **获得方式**：当命运值达到90以上时，玩家可选择"干预卡count + 1"来获得1张干预卡
- **持有数量**：玩家可以累积持有多张干预卡，每次选择该幸运操作时count增加1
- **使用时机**：玩家可在任意时机使用干预卡，不受命运值限制

**使用方法**：

- 玩家使用干预卡时，为角色X+1岁的事件预判栏目选择一个命运值区间（高度正面85~100、中等正面50~84、轻度正面1~49、轻度负面-1~-30、中等负面-31~-60、严重负面-61~-89），并为该区间填写一个自定义的事件描述
- 该描述将替换AI原本生成的该区间事件内容
- 实际触发仍需命运值落在玩家选择的区间（不保证触发）
- ⚠️ **约束**：不能在负面事件里写正面情节，或在正面事件里写负面情节
- 使用后，干预卡count减1

**生效机制**：
- AI需检查玩家提供的事件描述是否符合区间性质（正面/负面），如不符则拒绝执行
- 在本轮输出末尾的 JSON 代码块中，将玩家自定义的事件文本写入对应的槽位
- 实际触发仍需角色X+1岁时的命运值落在玩家选择的区间
- 干预卡为一次性消耗品，使用后立即失效

**显示格式**：
- 干预卡数量不由 AI 在输出面板中主动显示

### 5.5 宿命卡机制

**基本规则**：
- **获得方式**：当命运值达到90以上时，玩家可选择"宿命卡count + 1"来获得1张宿命卡
- **持有数量**：玩家可以累积持有多张宿命卡，每次选择该幸运操作时count增加1
- **使用时机**：玩家可在角色任意年龄使用宿命卡，不受命运值限制

**使用方法**：

- 玩家可使用"括号例外机制"向AI声明："（使用宿命卡，指定角色下一年龄强制触发【事件预判中的某个具体事件】）"
- 玩家需明确指定要触发的事件（可以是正面或负面事件）
- 使用后，宿命卡count减1

**生效机制**：
- 宿命卡生效的年龄，AI将无视玩家提交的原始随机数计算出的命运值
- 强制触发玩家指定的宿命事件
- 该年龄的命运值仍需正常计算并记录，但仅用于显示，不影响事件触发
- ⚠️ **失败事件例外**：宿命卡无法阻止失败事件。如果玩家提交的原始随机数计算后命运值落在失败事件区间（-100~-90），即使玩家使用了宿命卡，仍会触发失败事件并宣告游戏失败。宿命卡在失败事件面前自动失效且不会被消耗。
- 宿命卡为一次性消耗品，使用后立即失效（失败事件导致的自动失效除外）

**显示格式**：
- 宿命卡数量不由 AI 在输出面板中主动显示
- 使用宿命卡的年龄需特别标注："⚠️ 角色当前年龄使用宿命卡，强制触发指定事件"

```

📊 角色年龄：[X岁]（第X条记录）
👤 当前角色：[角色名称]（[如发生过角色切换，显示："第Y代继承者"]）
📋 最终命运值：[Z]
[如当前年龄使用宿命卡，显示："⚠️ 角色当前年龄使用宿命卡，强制触发指定事件"]

[如满足胜利条件，显示胜利类型："🏆 荣誉胜利"或"⚔️ 非荣誉胜利（角色切换次数：X次）"，并附上结局描述与胜利寄语]
```

## 7. 积分系统（⚠️ 所有计算由后台完成，AI 只读不算）

积分是对本局游戏表现的综合量化评估，保存于 `-total-score` snippet 中，游戏全程由后台程序自动维护，**AI 不得参与任何积分计算**。积分与游戏胜负条件无关，是独立的量化指标。

### 7.1 积分计算规则（供 AI 理解游戏机制，禁止自行套用）

| 触发事件 | 积分变化 |
|---------|---------|
| 每次推进番茄钟记录（move） | `总积分 += 本次最终命运值`（命运值为正则增加，为负则减少） |
| 超额健康度加成（每回合自动） | 当健康度 > 10 时，`总积分 += (健康度 - 10) × 10`（由助手技能等机制将健康度提升至10以上时触发，后台自动计算） |
| 成功达成阶段性节点里程碑 | `总积分 += 200` |
| Boss 战胜利（`is_boss_defeated` → true） | `总积分 += 800` |
| Boss 战失败（`is_boss_defeated` → false） | `总积分 -= 300`，然后 `总积分 × 0.9` |
| 游戏判负（任意失败来源触发） | `总积分 × 0.9` |
| 结算胜利（玩家点击"结算胜利"按钮） | `总积分 × 1.1` |
